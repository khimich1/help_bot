## 1. Цель

**Что строим:** семантический поиск по документам проекта — tool `search_project_knowledge` в существующем Telegram-боте.

**Зачем:** дизайнер на объекте спрашивает «что согласовали» без листания папки на 300+ файлов.

**Для кого:** те же 2 дизайнера (whitelist), не клиенты.

**Критерии успеха (срез 3.0):**

- 10 эталонных вопросов по Шкиперскому → ≥8 ответов с **верным источником** (ручной чеклист + pytest на mock index).
- Операционные фразы (`list_tasks`, `complete_task`, `set_active_project`) **не** вызывают search первым (≥9/10 на mock tool-calling).
- Ответы содержат: суть + путь к файлу + цитата ≤200 символов.
- При пустом результате — «в документах не нашёл», без выдуманных фактов.

## 4. Индексируемые документы (Tier 1)

| `doc_type` | Паттерн пути | Чанкование |
|------------|--------------|------------|
| `brief` | `Исходные материалы*/Общее*.pdf` | по секциям / ~800 токенов |
| `questionnaire` | `Исходные материалы*/Анкета*.pdf` | по комнатам / страницам |
| `site_report` | `**/Отчеты по выездам.docx` | по неделям выезда / абзацам |

**Metadata чанка:**

```json
{
  "project_name": "ЖК Шкиперский",
  "source_path": "Выезды авторский/Фото с выездов/Отчеты по выездам.docx",
  "stage": "Выезды авторский",
  "room": null,
  "doc_type": "site_report",
  "chunk_index": 0
}
```

`stage` / `room` — из пути (первый уровень папки; `Третий этап/N. Комната/` → room).

**Исключения при обходе:**

- Расширения не из списка: `.pdf`, `.docx` для Tier 1
- Путь содержит: `Неактуальное`, `старая`
- Файлы &lt; 50 символов извлечённого текста

**Не индексируем в 3.0:** jpg/png, webm, dwg, pln, psd, pdf-чертежи без текста (РД, планировки).

## 5. Tool: `search_project_knowledge`

| Поле | Тип | Описание |
|------|-----|----------|
| `query` | str | Поисковый запрос (из сообщения пользователя или уточнённый) |
| `project_id` | int, opt | ID проекта; если нет — активный проект пользователя |
| `room` | str, opt | Фильтр: «Кухня-гостиная», «Мастер-ванная» |
| `doc_type` | str, opt | `brief` \| `questionnaire` \| `site_report` |

**Выход (JSON):**

```json
{
  "ok": true,
  "project_name": "ЖК Шкиперский",
  "results": [
    {
      "text": "…фрагмент чанка…",
      "source_path": "Выезды авторский/…/Отчеты по выездам.docx",
      "doc_type": "site_report",
      "stage": "Выезды авторский",
      "room": null,
      "score": 0.82
    }
  ]
}
```

Пустой результат: `{"ok": true, "results": []}` — агент формулирует отказ сам.

## 6. Формат ответа пользователю (вариант C)

Порядок в сообщении Telegram:

1. **Суть** — 1–3 предложения, своими словами.
2. **Источник** — `Источник: {source_path}` (относительный путь в папке проекта).
3. **Цитата** — дословный фрагмент из `results[0].text`, до 200 символов, `…` при обрезке.

Если факты из 2+ чанков — до 2 источников, без дублирования цитат.

**Запрещено:** выдумывать факты, не подкреплённые `results`; ссылки на Я.Диск.

## 7. Когда вызывать search (D с разумными границами)

### Правило (в `prompt.py`)

> Если у пользователя **есть активный проект** (или проект назван в сообщении) и сообщение **не является чисто операционным** — **первым** вызови `search_project_knowledge`, передав суть запроса в `query`. Затем ответь пользователю или вызови другой tool.

### Search ПЕРВЫМ — да

| Пример | Почему |
|--------|--------|
| «Какой цвет дверей по Шкиперскому?» | вопрос о факте |
| «Что в брифе про кухню?» | вопрос о факте |
| «Закажи ту плитку в мастер-с/у до пятницы» | материал + задача → сначала факт |
| «Напомни пожелания Вики по гардеробной» | факт из анкеты/брифа |

### Search ПЕРВЫМ — нет

| Пример | Первый tool |
|--------|-------------|
| «Мои задачи на неделю» | `list_tasks` |
| «Закрой задачу 12» | `complete_task` (при необходимости предварительно `list_tasks`) |
| «Работаем по Петровым» | `set_active_project` |
| «Список проектов» | `list_projects` |
| «Привет» / «Спасибо» / «Ок» | без tool |
| «Создай проект Сидоровы» | `create_project` |

### Пустой RAG

Текст: «По документам проекта «ЖК Шкиперский» этого не нашёл. Уточни раздел (бриф / анкета / выезд) или посмотри на Диске.»

`create_tasks` — **только** если пользователь явно просит («создай задачу уточнить у клиента»).

## 8. Пользовательские сценарии

### 8.1 Справка по факту

```
Пользователь: По Шкиперскому какой цвет дверей?
→ search_project_knowledge(query="цвет дверей")
→ Ответ: Тёплый белый, ручки Лирика… + источник + цитата
```

### 8.2 Задача с отсылкой к решению

```
Пользователь: Закажи плитку в мастер-с/у как договаривались, до пятницы
→ search_project_knowledge(query="плитка мастер санузел")
→ create_tasks(project_id=…, tasks=[{title: "Заказать плитку Equipe Artisan…", due_date: …}])
→ «Создала задачу. Плитка: Equipe Artisan (из отчёта выезда).»
```

### 8.3 Операционка без search

```
Пользователь: Мои просроченные задачи
→ list_tasks(mine_only=true, overdue=true)
```

### 8.4 Не найдено

```
Пользователь: Какой ковёр в кабинете по Шкиперскому?
→ search → results=[]
→ «В документах не нашёл…» (без выдумки про ковёр)
```

## 13. Границы

**Всегда:**

- Тесты на extract, search, tool calling
- Секреты в `.env`
- Цитата + источник в ответах на факты
- Честный отказ при пустом RAG

**Никогда (срез 3.0):**

- OCR / vision по jpg и чертежам
- Яндекс.Диск API sync
- `list_project_files` и публичные ссылки
- Авто-`create_tasks` при пустом поиске
- Search перед `list_tasks` / `complete_task` / приветствиями
