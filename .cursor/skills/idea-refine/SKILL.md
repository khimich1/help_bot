# Idea Refine — идеи агентов

Партнёр по идеям. Превращает «хочу агента для …» в конкретный one-pager перед спекой.

## Когда применять

- Идея агента размыта («бот для поддержки», «умный помощник»)
- Неясно: сколько tools, какой граф, для кого
- Нужно сравнить 2–3 архитектурных направления
- Перед `spec-driven-development` — когда есть направление, но нет чёткого scope

**Не применять:** уже есть утверждённая спека; мелкий fix в существующем агенте.

## Три фазы

### Фаза 1: Понять и расширить (divergent)

1. **Переформулируй** как «Как мы можем …» (How Might We).
2. **3–5 вопросов** (на русском):
   - Для кого агент? (конечный пользователь / оператор / API)
   - Что считается успехом? (один диалог? автономное решение?)
   - Ограничения: модель, latency, бюджет токенов, on-prem?
   - Что пользователь делает сейчас без агента?
   - Почему агент, а не форма/скрипт?

3. **5–8 вариантов** через линзы (адаптировано под LangGraph):
   - **Инверсия:** агент только советует, не вызывает tools
   - **Убрать ограничения:** идеальный агент без лимита итераций
   - **Другая аудитория:** тот же домен, другой пользователь
   - **Упрощение:** один tool, один сценарий (как `airline_react_agent.py`)
   - **10x:** мульти-агент, субагенты, RAG
   - **Эксперт:** что очевидно для LLM-инженера, но не для заказчика

**В этом репозитории:** читай `airline_react_agent.py`, `.cursor/skills/langgraph-agent-development/` — варианты должны опираться на существующие паттерны.

Подробные линзы: [frameworks.md](frameworks.md). Критерии оценки: [refinement-criteria.md](refinement-criteria.md). Примеры: [examples.md](examples.md).

### Фаза 2: Оценить и сузить (convergent)

1. **Кластеризуй** откликнувшиеся идеи в 2–3 направления.
2. **Stress-test** каждое:
   - **Ценность:** боль vs «прикольно»; заменит ли текущий процесс?
   - **Реализуемость:** сколько tools, нужен ли RAG, сложность графа?
   - **Отличие:** почему не ChatGPT в чистом виде?
3. **Допущения** (явно):
   - На что ставим
   - Что убьёт идею
   - Что игнорируем в MVP

Будь честен, не yes-machine. Слабая идея — скажи мягко, но прямо.

### Фаза 3: Зафиксировать (one-pager)

```markdown
# [Название агента]

## Проблема (How Might We)
[одно предложение]

## Рекомендуемое направление
[2–3 абзаца: граф, tools, UX на русском]

## Допущения для проверки
- [ ] [допущение — как проверить]

## MVP
**В scope:** [1–3 tools, один happy path]
**Вне scope:** см. Not Doing

## Not Doing (и почему)
- [фича] — [причина отсечения]

## Открытые вопросы
- ...
```

Сохраняй в `docs/ideas/[имя].md` **только после подтверждения** пользователя.

## Следующий шаг

## Антипаттерны

- 20+ вариантов вместо 5–8 качественных
- Пропуск «для кого»
- План без Not Doing
- Игнор `airline_react_agent.py` как референса
- Сразу код без фаз 1–2

## Верификация

- [ ] How Might We сформулирован
- [ ] Пользователь и успех определены
- [ ] 2+ направления рассмотрены
- [ ] Допущения с способом проверки
- [ ] Not Doing явный
- [ ] Пользователь подтвердил направление
