# Примеры сессий idea-refine (агенты)

## Пример 1: Размытая идея

**Пользователь:** Хочу агента техподдержки для интернет-магазина

**Фаза 1:**

> **How might we** дать покупателю быстрый ответ по заказу и типовым вопросам без ожидания оператора?

Уточняющие вопросы:
1. Кто пишет — покупатель или оператор?
2. Есть API заказов или только FAQ?
3. Агент может отменять/менять заказы или только читать?

**Варианты (сокращённо):**
1. **FAQ-only** — один tool `search_faq`, без интеграции с заказами (MVP за день)
2. **Read-only orders** — `get_order_status` + FAQ (как `get_booking`)
3. **Full actions** — cancel, refund (нужен interrupt + auth)
4. **Инверсия** — агент готовит черновик ответа для оператора, не отвечает клиенту
5. **Упрощение** — фиксированные кнопки + LLM только для FAQ

**Фаза 2:** Вариант 2 — баланс ценности и реализуемости. Риск: LLM выдумает статус → только tool results в ответ.

**One-pager MVP:** 2 tools (`search_faq`, `get_order_status`), ReAct как в `airline_react_agent.py`, русский UX.

## Пример 2: Уже есть домен (авиа)

**Пользователь:** Расширить airline-агента — добавить отели

**Фаза 2 stress-test:**
- **Ценность:** cross-sell или отдельный продукт?
- **Реализуемость:** +3 tools vs отдельный граф?
- **Not Doing:** поиск отелей на карте, оплата — вне MVP

**Рекомендация:** отдельный `hotel_react_agent.py` с общим `core/tools_utils.py` — не раздувать один граф.

## Пример 3: Слабая идея (честный pushback)

**Пользователь:** Агент, который сам пишет код и деплоит в прод

**Ответ:** Высокий риск (LLM Top 10: excessive agency). MVP — агент только **предлагает** diff, deploy через CI после human approve. Not Doing: прямой deploy из tool.
