# Spec-Driven Development

Спецификация — общий источник правды. Код без спеки = угадывание.

## Когда применять

- Новый агент или значимая фича
- Неясные требования
- Затрагивает несколько файлов/модулей
- Архитектурное решение
- Работа > 30 минут

**Не применять:** однострочные fix, опечатки.

## Фазы (не переходи без валидации)

```
СПЕЦИФИКАЦИЯ → ПЛАН → ЗАДАЧИ → РЕАЛИЗАЦИЯ
      │           │        │          │
      ▼           ▼        ▼          ▼
  Пользователь проверяет каждую фазу
```

## Содержание спеки (6 блоков)

1. **Цель** — что строим, зачем, для кого, критерии успеха
2. **Команды** — как запускать, тестировать, линтить
   ```
   Запуск: python airline_react_agent.py
   Тесты: pytest tests/ -v
   ```
3. **Структура** — где код, тесты, конфиг
4. **Стиль кода** — пример сниппета из проекта
5. **Тестирование** — pytest, что покрывать, уровни
6. **Границы**
   - Всегда: тесты, `.env` для секретов, русский UX
   - Спросить: новые зависимости, смена модели
   - Никогда: секреты в git, удаление тестов без согласия

## Шаблон спеки

```markdown
# Спека: [название агента/фичи]

## Цель
...

## Пользовательские сценарии
1. ...
2. ...

## Tools (если агент)
| Tool | Описание | Вход | Выход |

## Нефункциональные требования
- Язык ответов: русский
- Лимит итераций: ...
- Модель: из .env

## Критерии приёмки
- [ ] ...
- [ ] ...

## Вне scope
- ...
```

## Правила

1. Сначала допущения — потом текст спеки
2. Спека на **русском**
3. Не пиши код до утверждения спеки пользователем
4. После спеки → `planning-and-task-breakdown`
